## Why
The bot was functional but opaque. You could log expenses and query totals via commands, but there was no way to browse, edit, or review what the bot had parsed. A receipt photo might get parsed as "coffee 4.50" when it was actually "lunch 14.50" — and you'd never know unless you checked totals and the numbers felt off.

Telegram Mini Apps solve this elegantly: they open inside the Telegram app itself, inherit the user's auth context, and feel native. No separate login, no App Store, no URL to remember.

## Key Decisions
| Decision | Options Considered | Chosen | Why |
|----------|-------------------|--------|-----|
| App delivery | Separate web app, PWA, Telegram Mini App | Telegram Mini App | Native Telegram integration, no separate auth flow, opens inline. The user is already in Telegram — don't make them leave. |
| Frontend framework | React, Svelte, Vue, vanilla | React 19 | Broader ecosystem, easier to find components. Svelte would've been lighter but I'm faster in React and this is a side project where velocity matters. |
| Component library | Radix primitives, Material UI, shadcn/ui | shadcn/ui | Copy-paste components, not a dependency. Full control over styling. Tailwind-native. Perfect for rapid prototyping where you'll customize everything later anyway. |
| Auth | Custom JWT, session cookies, Telegram initData HMAC | Telegram initData HMAC | Telegram provides a signed payload with every Mini App launch. Validate the HMAC server-side and you have authenticated user identity with zero user friction. |
| Hosting | Cloudflare Pages, Vercel, same Worker | Cloudflare Pages (via wrangler) | Static assets on Pages, API on Workers. Same ecosystem, same deploy pipeline. |

## How (Workflow)
This was a two-evening sprint. First evening: scaffold the frontend, get Vite building, wire up shadcn, and implement the API + auth layer. Second evening: build the actual UI screens and connect them to the API.

The auth implementation was the most interesting part. Telegram Mini Apps pass `initData` — a query string signed with the bot token using HMAC-SHA-256. Server-side validation means you can trust `user.id` from the payload without any separate login flow. Wrote tests for this immediately because getting auth wrong is not something you want to discover in production.

## Learnings
- **Telegram Mini App auth is elegant but underdocumented.** The HMAC validation flow is straightforward once you understand it, but the official docs bury the important details. The key insight: you hash the bot token with "WebAppData" as the HMAC key, then use that to verify the initData signature.
- **shadcn/ui is the right call for prototyping.** You get reasonable-looking components immediately, and because they're copied into your project (not imported from a package), you can rip them apart later without fighting a library's opinion.
- **The Review Queue was more important than the Dashboard.** I built the Dashboard first because it felt like the obvious screen, but the real value was the Review Queue — letting users confirm or fix what the AI parsed. This is where trust in the system gets built.
- **API design for Mini Apps is just REST.** Nothing special needed. The only Telegram-specific part is the auth header. Everything else is standard CRUD with JSON responses.
